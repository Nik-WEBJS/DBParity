# DBParity

**Докажите, что при миграции БД не потерялась и не исказилась ни одна строка.**

[English version →](README.md)

DBParity — верификатор эквивалентности данных при миграциях
(Oracle / MSSQL / SQLite → PostgreSQL). Читает обе базы потоково в порядке
первичного ключа, сравнивает строка за строкой с нормализацией «ловушек
миграции» и генерирует отчёт, который интегратор кладёт заказчику на стол
при сдаче проекта.

> Статус: **v0.1 alpha**. Ядро и PostgreSQL-адаптер проверены (в том числе
> на живом PostgreSQL 18). Oracle-адаптер написан, но не обкатан в бою —
> **очень нужны тестеры с реальными Oracle-инстансами**, открывайте issue.

## Зачем

Миграции проваливаются не на конвертации кода, а на сверке данных.
Gartner прогнозирует провал >70% проектов ухода с мейнфреймов; миграция
ERP Бирмингема выросла с £19M до £144M+ — сломалась именно сверка.
Инструментов конвертации схем и SQL много; тех, кто *доказывает, что
данные доехали целыми*, — почти нет.

## Что ловит

Реальные расхождения — потерянные и лишние строки, изменённые значения,
дубли PK, дрейф схемы (колонки, типы, PK) — и при этом **не** флагует то,
что лишь выглядит по-разному: Oracle `''`==NULL, `1.50` vs `1.5`,
шум float (epsilon), таймзоны (UTC), Oracle DATE с временем vs date,
CHAR-паддинг, юникод NFC, `0/1/Y/N` vs boolean, BLOB по MD5, точность
timestamp. Полная таблица — в [английском README](README.md) и PLAN.md §3.

## Быстрый старт

```bash
pip install -e ".[postgres]"        # из клонированного репозитория
dbparity demo --outdir demo_out     # демо со специально внесёнными расхождениями
# → откройте demo_out/dbparity_report.html
dbparity compare -c config.yaml     # реальная сверка
```

Пример `config.yaml` — в английском README (структура: source/target,
tables, pk_overrides, exclude_columns, rules, mask_values, report).

### Коды выхода

`0` — эквивалентно · `1` — найдены расхождения · `2` — ошибка запуска.
Удобно как обязательный шаг CI/CD перед переключением трафика.

## Производительность

Потоковый merge: O(n) по времени, O(batch) по памяти — таблица целиком
в RAM не сидит. `python3 bench/bench.py 1000000` (1 млн строк × 2 стороны):
**~400 тыс. строк/с** с fast-path по типам (~310 тыс. generic).
Дальше по роадмапу — сегментные хэши на стороне БД.

## Тесты

```bash
pip install -e ".[dev,postgres]"
pytest tests/ -v

# интеграция с живым PostgreSQL: docker compose up -d, затем
DBPARITY_PG_DSN="host=127.0.0.1 port=5432 dbname=dbparity user=postgres password=dbparity" \
  pytest tests/test_postgres_integration.py -v
# без Docker: node scripts/pglite_server.mjs (PGlite — Postgres в WASM)
```

## Роадмап

Сегментные DB-side хэши для таблиц 100M+ строк · parallel-run ·
resume после обрыва · обкатка Oracle/MSSQL · веб-консоль поверх ядра.

Архитектура: [PLAN.md](PLAN.md).
