# DBParity — план разработки MVP

**Продукт:** инструмент верификации миграций БД — доказывает эквивалентность данных
источника (Oracle/MSSQL) и приёмника (PostgreSQL) и генерирует отчёт для заказчика.
**Покупатель:** интеграторы, ведущие миграционные проекты (СНГ: КИИ/импортозамещение → ЕС: Oracle-exit, SAP 2027).

## 1. Технологии и инструменты

| Слой | Выбор | Почему |
|---|---|---|
| Язык | Python 3.10+ | скорость разработки, зрелые DB-драйверы, достаточно для стриминга |
| Драйверы | `oracledb` (thin, без Instant Client), `psycopg3`, `pyodbc` (MSSQL, опционально), `sqlite3` (встроен — демо/тесты) | все опциональны: ядро не зависит от драйверов |
| Конфиг | YAML (`pyyaml`) | человекочитаемые сценарии сверки |
| Отчёт | Jinja2 → один самодостаточный HTML | интегратор пересылает заказчику одним файлом |
| UI-кит отчёта | **Tabler** (Bootstrap 5, MIT, 40k+ stars) через CDN + Chart.js | лучший open-source кит для data-heavy дашбордов в 2026 |
| CLI | argparse + `rich` | нулевой фреймворк-налог, красивый консольный вывод |
| Тесты | `pytest` | стандарт |

## 2. Архитектура кода

Гексагональная: диалект-независимое ядро + тонкие адаптеры по краям.

```
dbparity/
├── cli.py                 # точки входа: compare, demo
├── config.py              # загрузка/валидация YAML-конфига
├── adapters/
│   ├── base.py            # интерфейс Adapter: schema(), stream_rows(), row_count()
│   ├── sqlite_adapter.py  # демо и тесты
│   ├── postgres_adapter.py# psycopg3, keyset-пагинация
│   ├── oracle_adapter.py  # python-oracledb thin mode
│   └── mssql_adapter.py   # pyodbc (заглушка-опция)
├── core/
│   ├── models.py          # dataclass'ы: TableResult, RowDiff, RunResult…
│   ├── normalize.py       # правила эквивалентности типов (см. §3)
│   ├── compare.py         # стриминговый merge-diff двух потоков по PK
│   └── schema_diff.py     # сверка структуры: таблицы, колонки, типы, PK
├── report/
│   ├── render.py          # Jinja2 → HTML
│   └── templates/report.html.j2
├── demo/seed.py           # две демо-БД с заранее известными расхождениями
└── tests/                 # юнит + интеграционные
```

**Поток данных:** `config.yaml → [Adapter src] + [Adapter dst] → schema_diff →
для каждой таблицы: два потока строк, отсортированных по PK → normalize →
merge-diff → TableResult → RunResult → render (HTML + JSON) + exit-code`.

**Алгоритм сравнения** — merge двух упорядоченных по PK потоков, O(n) по времени,
O(batch) по памяти (сервер отдаёт строки чанками, keyset-пагинация, без OFFSET):
- `pk_src < pk_dst` → строка потеряна в приёмнике (`missing_in_target`)
- `pk_src > pk_dst` → лишняя строка в приёмнике (`extra_in_target`)
- равны → поколоночное сравнение нормализованных значений → `mismatch` с детализацией

Примеры расхождений сохраняются с лимитом (по умолчанию 50 на категорию),
счётчики — полные. Значения в отчёте можно маскировать (`mask_values: true`).

## 3. Правила нормализации (ядро ценности — «ловушки миграций»)

| Ловушка | Правило |
|---|---|
| Oracle: `''` == `NULL` в VARCHAR2 | при source=oracle пустая строка → NULL |
| `NUMBER` vs `NUMERIC`: `1.50` vs `1.5` | сравнение через Decimal, нормализация хвостовых нулей |
| Float | сравнение с настраиваемым epsilon |
| `DATE` Oracle несёт время | опция truncate_time_if_date_target |
| Таймзоны | приведение к UTC перед сравнением |
| `CHAR(n)` паддинг пробелами | опция rtrim |
| Юникод | нормализация NFC |
| Булевы: `0/1/'Y'/'N'` vs `bool` | таблица маппинга |
| BLOB/bytea | сравнение по MD5 |
| Точность timestamp (мкс vs нс) | усечение до общей точности |

## 4. Обвёртка (шаг 2 задания)

Отчёт — витрина продукта: вердикт-баннер (EQUIVALENT / NOT EQUIVALENT + % совпадения),
KPI-карточки (строк проверено, расхождений, таблиц OK), donut общего результата,
bar-chart расхождений по таблицам, схемные различия, разворачиваемые примеры
расхождений с подсветкой отличающихся колонок. Стиль — Tabler CDN, тёмная/светлая тема.
Плюс аккуратный консольный вывод (rich): таблица результатов + вердикт.

## 5. Тестирование (шаг 3 задания)

- Юнит: normalize (каждая ловушка), compare (все категории расхождений: missing/extra/mismatch/дубли PK), schema_diff, config.
- Интеграционные: демо-прогон end-to-end на sqlite-парах → ожидаемые счётчики расхождений, валидный HTML, корректный exit-code (0 — эквивалентно, 1 — расхождения).
- Ограничение песочницы: Oracle/MSSQL интеграционно не тестируются (нет серверов) — адаптеры покрыты контрактными тестами через общий интерфейс; sqlite-адаптер эмулирует диалектные флаги.

## 6. Ограничения MVP и роадмап

MVP сознательно не делает: parallel-run, генерацию тестов из трафика, сравнение
performance-планов, распределённый режим. Роадмап: сегментные DB-side хэши
(сравнение агрегат-хэшей PK-диапазонов, детализация только при расхождении) →
parallel-run → SAP→1С коннекторы → веб-консоль поверх ядра.

## 7. Статус закалки (выполнено после MVP)

- **Live-PostgreSQL интеграция**: psycopg-адаптер проверен на живом PostgreSQL 18
  по wire-протоколу (в песочнице — PGlite + pglite-socket; локально — docker compose).
  Интроспекция, server-side курсоры, numeric/boolean/timestamptz/date — без ложных
  расхождений на кросс-движковых ловушках; опция `server_side: false` для сред
  без DECLARE CURSOR.
- **Fast-path нормализации**: пер-колоночные прекомпилированные функции по
  логическим типам схемы (обход isinstance-цепочки), фолбэк на generic при
  неожиданном типе. Паритет с generic закреплён тестом. Бенчмарк на 1 млн строк
  × 2 стороны: 310 → 400 тыс. строк/с (×1.3), `bench/bench.py`.
